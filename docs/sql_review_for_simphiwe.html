What you got right

You're using the right tables. int_customer_category_spend, stg_customers, and mart_cluster_output — that's the exact stack we built for audience profiling. You also remembered to join stg_customers for demographics, which is the bit a lot of people skip. And the structure of the analysis (size → demographics → segment mix → co-shopping) is the right shape. So the thinking is solid. The execution needs three tightens.

What to fix

1You're scanning the same big table 5 times

Every block of your script does its own FROM int_customer_category_spend cs JOIN stg_customers c. That's a 3.6 billion row table. Each time you run a query, BigQuery scans it again. Five queries = five scans = five times the cost.

And every block re-types the same 16-brand WHERE DESTINATION IN (...) list. If you drop a brand or fix a typo, you have to change it in 16 places.

Symptom you can see in the BQ console: next to the "Run" button there's a "This query will process X MB" estimate. If your 5 queries each say ~400 MB, you're paying ~2 GB for what could be ~400 MB if you did it once.

Fix: use a CTE. Define the brand list and the audience once at the top, then reuse them downstream.

2Your co-shopping filter has a bug

In the "where else do these customers shop" query you wrote:

WHERE cs2.DESTINATION != 'VOX'   -- excluding vox

That excludes VOX but still includes Afrihost, MWEB, Telkom, RAIN, Vumatel — all the other internet brands. So your "co-shopping" results will show that internet customers also shop at... other internet brands. Which is true but not useful.

Why this matters: the whole point of a co-shopping query is "what do these people buy outside the category I'm pitching for". A media planner reading this output will see "internet customers also love Afrihost!" and think you found an insight, when actually you just found the audience definition you yourself wrote three lines up.

Fix: exclude the whole brand list, not just VOX.

WHERE cs2.DESTINATION NOT IN (
    'VOX', 'AFRIHOST', '3 NET NINE NINE', 'ZA DOMAINS', 'AXXESS',
    'COOL IDEAS', 'MWEB', 'TELKOM', 'EXACTTA GROUP SRL', 'VUMATEL',
    'G CONNECT', 'ACCELERIT TECHNOLOGIES', 'RAIN (DATA)', 'IKEJA',
    'WEB AFRICA', 'SUPERSONIC'
)

Even better — if you put the brand list in a CTE (Fix #1), it becomes:

WHERE cs2.DESTINATION NOT IN (SELECT brand FROM brands)

Now you maintain the list in one place and use it twice (once to define the audience, once to exclude from co-shopping).

3You changed the audience definition halfway through the script

Look at your own results:

Three different populations. They all live in the same file. If you slide a chart from one section into a deck and quote the wrong number, the slide is wrong. Worse, you won't catch it because all three queries "look like" they're profiling the same audience.

This bit me too when I started. The fix isn't to never narrow — narrowing is exactly the point of a brand-specific audience. The fix is to name your audiences and stick to one per profile section.

Fix: if you need multiple slices, give each one a name and a comment block:

-- AUDIENCE A: All internet shoppers (the base)
-- 347,697 customers
WITH audience_all AS ( ... )

-- AUDIENCE B: Female 18-45, mid-income (narrow for Revlon-style pitch)
-- 79,915 customers
WITH audience_female_midmarket AS ( ... )

Then in your output, label each rollup by which audience it came from. Future-you will thank present-you when someone says "wait, which number is the right one?"

4Dead code at the bottom

Lines 217 onwards have a stray WHERE cs.DESTINATION IN (...) with no SELECT attached. Leftover from copy-paste. Delete it. If you ever paste this script into someone else's editor they'll think it's part of the logic.

When you're done with a script, scroll to the bottom and delete anything that doesn't run. Treat your SQL like real code, not a scratchpad.

How to do it — the clean version

Same analysis, in one query. Brand list defined once, audience defined once, every rollup runs off the same enriched CTE so nothing rescans the 3.6B-row table.

AThe single audience profile query

-- ─────────────────────────────────────────────────────────────────
-- Internet & Data audience profile
-- ONE definition, ONE scan, all profile dimensions in one output.
-- ─────────────────────────────────────────────────────────────────

WITH
-- Define brands ONCE
brands AS (
    SELECT brand FROM UNNEST([
        'VOX', 'AFRIHOST', '3 NET NINE NINE', 'ZA DOMAINS', 'AXXESS',
        'COOL IDEAS', 'MWEB', 'TELKOM', 'EXACTTA GROUP SRL', 'VUMATEL',
        'G CONNECT', 'ACCELERIT TECHNOLOGIES', 'RAIN (DATA)', 'IKEJA',
        'WEB AFRICA', 'SUPERSONIC'
    ]) AS brand
),

-- Define the audience ONCE (distinct customers + their spend)
audience AS (
    SELECT
        cs.UNIQUE_ID,
        SUM(cs.dest_spend) AS spend_in_category
    FROM `fmn-sandbox.analytics.int_customer_category_spend` cs
    JOIN brands b ON cs.DESTINATION = b.brand
    GROUP BY cs.UNIQUE_ID
),

-- Attach demographics + behavioural segment ONCE
enriched AS (
    SELECT
        a.UNIQUE_ID,
        a.spend_in_category,
        c.age,
        c.age_group,
        c.estimated_income,
        c.income_group,
        c.gender_label,
        co.segment_name
    FROM audience a
    LEFT JOIN `fmn-sandbox.staging.stg_customers` c
        ON a.UNIQUE_ID = c.UNIQUE_ID
    LEFT JOIN `fmn-sandbox.marts.mart_cluster_output` co
        ON a.UNIQUE_ID = co.UNIQUE_ID
)

-- All profile slices come from `enriched`. No re-scanning.

SELECT
    'overall' AS slice,
    COUNT(*) AS audience_size,
    ROUND(SUM(spend_in_category), 0) AS total_spend,
    ROUND(AVG(spend_in_category), 0) AS avg_spend,
    ROUND(AVG(age), 1) AS avg_age,
    ROUND(AVG(estimated_income), 0) AS avg_income,
    ROUND(COUNTIF(gender_label = 'Female') * 100.0 / COUNT(*), 1) AS pct_female
FROM enriched

UNION ALL

-- Age-group breakdown
SELECT
    CONCAT('age:', age_group),
    COUNT(*),
    ROUND(SUM(spend_in_category), 0),
    ROUND(AVG(spend_in_category), 0),
    ROUND(AVG(age), 1),
    ROUND(AVG(estimated_income), 0),
    ROUND(COUNTIF(gender_label = 'Female') * 100.0 / COUNT(*), 1)
FROM enriched
WHERE age_group IS NOT NULL AND age_group != 'Unknown'
GROUP BY age_group

UNION ALL

-- Segment breakdown (Champions, Loyal High Value, At Risk, etc.)
SELECT
    CONCAT('segment:', segment_name),
    COUNT(*),
    ROUND(SUM(spend_in_category), 0),
    ROUND(AVG(spend_in_category), 0),
    ROUND(AVG(age), 1),
    ROUND(AVG(estimated_income), 0),
    ROUND(COUNTIF(gender_label = 'Female') * 100.0 / COUNT(*), 1)
FROM enriched
WHERE segment_name IS NOT NULL
GROUP BY segment_name

ORDER BY slice;
One scan. One run. Every dimension you were profiling separately now comes out as one tidy table. Easier to read, easier to paste into a deck, cheaper to run.

BThe co-shopping query (separately, because it needs a different join shape)

WITH
brands AS (
    SELECT brand FROM UNNEST([
        'VOX', 'AFRIHOST', '3 NET NINE NINE', 'ZA DOMAINS', 'AXXESS',
        'COOL IDEAS', 'MWEB', 'TELKOM', 'EXACTTA GROUP SRL', 'VUMATEL',
        'G CONNECT', 'ACCELERIT TECHNOLOGIES', 'RAIN (DATA)', 'IKEJA',
        'WEB AFRICA', 'SUPERSONIC'
    ]) AS brand
),
audience AS (
    SELECT DISTINCT cs.UNIQUE_ID
    FROM `fmn-sandbox.analytics.int_customer_category_spend` cs
    JOIN brands b ON cs.DESTINATION = b.brand
)
SELECT
    cs2.CATEGORY_TWO,
    cs2.DESTINATION,
    COUNT(DISTINCT cs2.UNIQUE_ID) AS audience_shoppers,
    ROUND(SUM(cs2.dest_spend), 0) AS audience_spend
FROM audience a
JOIN `fmn-sandbox.analytics.int_customer_category_spend` cs2
    ON a.UNIQUE_ID = cs2.UNIQUE_ID
LEFT JOIN brands b ON cs2.DESTINATION = b.brand
WHERE b.brand IS NULL    -- exclude ALL internet brands, not just VOX
GROUP BY cs2.CATEGORY_TWO, cs2.DESTINATION
ORDER BY audience_spend DESC
LIMIT 50;
The LEFT JOIN brands b ... WHERE b.brand IS NULL trick is called an anti-join. It keeps rows from cs2 that have no match in brands — i.e. everything that isn't an internet brand. Cleaner than a giant NOT IN list, especially as the brand list grows.

Patterns to remember

Audience profile pattern (memorise this shape):
  1. brands CTE — define your "this is what I'm pitching for" list ONCE.
  2. audience CTE — pick customers who match. Just UNIQUE_ID (+ optionally their category spend).
  3. enriched CTE — left-join demographics from stg_customers and segment from mart_cluster_output.
  4. Every rollup (size, demographics, segments, age groups, geography) selects from enriched. No re-scanning.
When you want to slice the audience differently:
  • Don't change the filter inside one big script and risk confusing yourself.
  • Do define multiple named audience CTEs: audience_all, audience_female_midmarket, audience_premium. Each profile section is explicit about which one it's measuring.
When you're excluding something:
  • If you're excluding one thing, != is fine.
  • If you're excluding a group, use NOT IN (...) or — better — an anti-join (LEFT JOIN ... WHERE x IS NULL).
  • Always ask yourself: "is the thing I'm excluding the whole category, or just one entry in it?" That's exactly the trap you hit with the != 'VOX' bug.
Before you send a script to someone:
  1. Scroll to the bottom — delete any dead code.
  2. Read the comments — do they still match what the code does?
  3. Run the whole thing top to bottom in a fresh tab — does it execute end-to-end without errors?
  4. Check the "bytes processed" estimate next to Run — is it bigger than you'd expect? That usually means you're re-scanning.

Summary

What I want you to take from this:

1. CTEs aren't optional for analyses with multiple rollups. They're how you avoid paying for the same scan five times.

2. Audiences need names. If a script profiles different populations, label them. Don't let the reader (or future-you) guess.

3. Exclusion lists must match definition lists. The single biggest source of "we got the wrong number" in audience work is excluding one brand but forgetting its siblings.

4. Your approach was on the right track — these are tightening fixes, not a rebuild. Next time, start with the CTE skeleton above and fill it in. You'll get to the same answers in a quarter of the code.
