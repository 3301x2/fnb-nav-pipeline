What you got right

You're profiling an audience the right way: size → spend → demographics → segments → co-shopping. That's the textbook shape. You're joining demographics from customer data, attaching behavioural segments, looking at what else they buy. The thinking is solid. The execution has two big issues worth fixing now so they don't become habits.

The big rethink: who's the audience?

Your script defines the audience as "people who shop at any of 16 internet brands" — VOX, Afrihost, MWEB, Telkom, RAIN, all of them. That's not a VOX audience. That's an "internet category" audience. If your goal is to help VOX pitch for ad spend, the audience is VOX's actual customers, not their competitors' customers.

The 16-brand list isn't useless — it's the competitive set for benchmarking. Two different things:

If your question is...Then the audience is...
"Who are VOX's customers? What do they look like? Who else should pitch them?"DESTINATION = 'VOX'
The other brands are for comparing VOX against, not for inflating the audience size.
"What's the total addressable market in the Internet category?"CATEGORY_TWO = 'Internet'
This is what your current script returns. Useful for sizing the category, not for a VOX-specific pitch.
"Who shops in the category but NOT at VOX?" (poaching targets)CATEGORY_TWO = 'Internet' AND UNIQUE_ID NOT IN (VOX customers)
Why this matters in the meeting: if you go in saying "your audience is 347,697 customers, age 41, R33k income, 46% female" — that's the average internet customer in SA, not the VOX customer. The VOX team can already get that from a press release. What they actually want is "your customers are X, the competitive set is Y, here's the gap, here's who to poach." Two queries, not one.

Other fixes

2Co-shopping query has a subtle bug

You wrote:

WHERE cs2.DESTINATION != 'VOX'   -- excluding vox

That excludes VOX but still includes Afrihost, MWEB, Telkom, RAIN, Vumatel — all the other internet brands. So your "co-shopping" results will say "VOX customers also love Afrihost!" which is just the audience definition leaking back into the output.

The clean version — using the marts

Same analysis, in 5 small focused queries. Each one is single-purpose and easy to read.

AVOX market position vs the competitive set

Use mart_destination_benchmarks. No joins, no customer-level scans. One small table.

SELECT
    DESTINATION,
    customers,
    total_spend,
    avg_txn_value,
    spend_per_customer,
    market_share_pct,
    penetration_pct,
    spend_rank
FROM `fmn-sandbox.marts.mart_destination_benchmarks`
WHERE CATEGORY_TWO = 'Internet'
ORDER BY spend_rank;
Bytes processed (dry-run): under 1 MB. This is the table I built for exactly this question. Always confirm via the "Bytes processed" estimate above the Run button.

BInternet-category demographics (everyone in the category, not VOX-only)

Use mart_demographic_summary. Pre-aggregated by category × demographic group.

SELECT
    gender_label,
    age_group,
    income_group,
    customers,
    total_spend,
    avg_spend_per_customer
FROM `fmn-sandbox.marts.mart_demographic_summary`
WHERE CATEGORY_TWO = 'Internet'
ORDER BY customers DESC;
Heads up: this is at the CATEGORY level. It tells you the demographics of "everyone shopping in Internet", not "VOX customers specifically". For VOX-only demographics, see Query C.

CVOX-specific customer demographics

No mart exists at the per-destination demographic level (yet). So we filter the raw table to just VOX UNIQUE_IDs in a CTE, then join stg_customers.

WITH vox_customers AS (
    SELECT DISTINCT UNIQUE_ID
    FROM `fmn-sandbox.analytics.int_customer_category_spend`
    WHERE DESTINATION = 'VOX'     -- only VOX, not the whole category
)
SELECT
    c.age_group,
    c.income_group,
    c.gender_label,
    COUNT(*) AS customers,
    ROUND(AVG(c.age), 1) AS avg_age,
    ROUND(AVG(c.estimated_income), 0) AS avg_income
FROM vox_customers v
JOIN `fmn-sandbox.staging.stg_customers` c
    ON v.UNIQUE_ID = c.UNIQUE_ID
GROUP BY c.age_group, c.income_group, c.gender_label
ORDER BY customers DESC;
Honest cost note: the table is clustered by CATEGORY_TWO + DESTINATION which helps when you filter directly. But when the engine has to do a CTE-then-join shape like this, it can't always prove which clusters it can skip. A dry-run of this query against ~213M rows shows around 10 GB processed. That's still way cheaper than your 5-query script, but it's not free. Always check "Bytes processed" before pressing Run.

DVOX customers' behavioural segment mix

Use mart_client_segment_mix. I built this exactly for "what does the segment mix look like for a specific client".

SELECT
    segment_name,
    segment_customers,
    pct_of_client_customers,
    pct_of_client_spend,
    fnb_pct_of_customers,
    index_vs_fnb
FROM `fmn-sandbox.marts.mart_client_segment_mix`
WHERE DESTINATION = 'VOX'
ORDER BY pct_of_client_customers DESC;
The index_vs_fnb column is gold for your pitch: it tells you whether VOX over- or under-indexes on each segment vs the FNB-wide average. "Your Champions are 2× the FNB average" is a way better headline than "You have X champions". See the Real Findings section below — the numbers from this query are striking.

ECo-shopping (where else do VOX customers spend)

This one genuinely needs to scan the raw table — because we want all categories, not just internet. We limit the scan by picking the VOX audience first as a CTE.

WITH vox_customers AS (
    SELECT DISTINCT UNIQUE_ID
    FROM `fmn-sandbox.analytics.int_customer_category_spend`
    WHERE DESTINATION = 'VOX'
)
SELECT
    cs.CATEGORY_TWO,
    cs.DESTINATION,
    COUNT(DISTINCT cs.UNIQUE_ID) AS vox_shoppers,
    ROUND(SUM(cs.dest_spend), 0) AS vox_spend
FROM vox_customers v
JOIN `fmn-sandbox.analytics.int_customer_category_spend` cs
    ON v.UNIQUE_ID = cs.UNIQUE_ID
WHERE cs.CATEGORY_TWO != 'Internet'   -- exclude the whole competitive category, not just VOX
GROUP BY cs.CATEGORY_TWO, cs.DESTINATION
ORDER BY vox_spend DESC
LIMIT 50;
The single CATEGORY_TWO != 'Internet' excludes VOX AND all 15 competitors in one shot — no giant NOT IN list, and it auto-handles any new internet brands we add later.

Important caveat — be honest about what the segments actually mean:

The K-Means model is trained on each customer's FNB-wide behaviour, not their VOX-specific behaviour. So "Champion" = "top FNB spender", not "top VOX spender". The 210 / 401 indexes are telling you "VOX customers over-represent FNB's top-tier spenders" — which is genuinely useful and pitchable, but it's not the same as "VOX has its own internal Champions vs Dormants based on how much they spend with VOX".

If we ever want a VOX-internal segmentation (e.g., to differentiate VOX's heavy users from their occasional ones), we'd need to train a separate model on VOX-only features. For now, the global segment mix is the most useful thing we have — just frame it accurately in the pitch.
Why this still matters: a generic "47% female, age 41" profile is what every audience deck looks like. "Your customers over-index 4× on FNB's Loyal High Value tier" is unique to VOX, demonstrably true from our data, and tells VOX something they can't read in a press release. That's the differentiator — as long as you frame it as "your base over-represents FNB's premium customers" and not "we segmented VOX's own customers".

Patterns to remember

1. Check the marts first. Before writing SQL against int_* or stg_*, look at what's in marts.*. If a mart already answers your question, use it. The raw tables are ~213M rows; the marts are kilobytes-to-MB.
2. Know the difference between "client" and "competitive set".
  • The client is your audience: WHERE DESTINATION = 'VOX'
  • The competitive set is what you benchmark against, not what you add to the audience
  • Don't bundle them together unless you genuinely want "the whole category" (which is a different question)
3. Clustering helps, but check the dry-run. When a table is clustered (like int_customer_category_spend by CATEGORY_TWO + DESTINATION), filtering on those columns is usually cheaper than filtering on something else. But complex queries with CTEs and joins don't always benefit — BigQuery sometimes can't prove which clusters it can skip. Always look at "Bytes processed" next to the Run button before you actually run.
4. Name your audiences. If your script profiles multiple slices, give each one a name and a comment. Never let a reader (including future-you) guess "wait, which audience is this section about?"
5. "Bytes processed" is the actual safety rail. BigQuery shows it next to the Run button.
  • Mart queries: sub-100 MB
  • CTE + customer-level joins: single-digit GB is normal
  • Double-digit GB = pause and rethink before pressing Run
6. Exclude by category, not by enumeration. If you're excluding "all internet brands", write CATEGORY_TWO != 'Internet' instead of listing 16 brand names. Cleaner, future-proof when new entrants join the category.

Summary

Three things to take away:

1. VOX = the client. The competitive set isn't part of the audience — it's what you compare against. Don't union them into the audience definition.

2. Use the marts. mart_destination_benchmarks, mart_demographic_summary, mart_client_segment_mix answer most pitch questions without ever touching the large fact table. That's why they exist.

3. When you do need the raw table, scope the scan with a CTE first (pick the customer set, then join). Watch the dry-run cost — clustering doesn't always rescue you when there's a join.

Your shape is right. Your instinct to attach demographics + segments is right. The fixes are just about which tables to reach for and how to define the question. Hit me back if anything's unclear.
