Looker Studio Build Guide v2 — April 2026

Build one robust NAV Analytics dashboard that covers every pitch scenario. Supersedes the earlier three guides. ~3 hours if you follow it end-to-end.

Cost note: This build runs no ML training and no full-data scans. The only BigQuery work is Step 4 of the pipeline (~R10 once) plus the SQL views (R0 — views don't store data). Do not run bash scripts/run.sh without a step argument — that reprocesses everything.

What you're building

One Looker Studio report, six pages, driven by two global filters (Client and Category). Every page adapts to those filters, so the analyst picks a client once and every chart updates.

SegmentsPer-client segment distribution + definitionsv_client_segment_mix + v_dashboard_segments

Key fix in this version: The Segments page now uses v_client_segment_mix, which returns different numbers for Clicks vs Pick n Pay vs any other client. The old build pulled from mart_cluster_output un-filtered, so every client looked identical. If you're updating an existing report, Step 3 below is the one that matters.

Prerequisites — run on the sandbox machine first

Pull the latest pipeline code

cd ~/fnb-nav-pipeline
git pull

Build the new marts (Step 4 only — cheap)

Runs the existing mart layer plus the two new tables: mart_client_segment_mix and mart_audience_client_overlap. No ML retraining, no staging rebuild.

bash scripts/run.sh sandbox 4
Expected cost: under R10. You are not scanning the 2.2B-row transactions table — these marts read from already-built intermediate tables.

Build / refresh the Looker views (free)

bash scripts/run.sh sandbox 6

Views store zero data — they're just saved queries. Refreshing them costs nothing. This step creates the new v_client_segment_mix, v_audience_catalog, and v_audience_client_overlap alongside the existing 19 views.

Verify the new objects exist

bq ls fmn-sandbox:marts | grep -E "mart_client_segment_mix|mart_audience_client_overlap|v_client_segment_mix|v_audience_catalog|v_audience_client_overlap"

You should see all five names. If any are missing, re-run the step that creates them.

Open Looker Studio

Go to lookerstudio.google.com, sign in with your FNB Google account, click + Create → Report. Name it NAV Analytics Dashboard.

Add a data source manually (do once, repeat 10× — or use the shortcut above)

Click Add data → BigQuery → authorise → My projects → fmn-sandbox → marts → pick the view → Add. Rename it to match the view name exactly so later steps are unambiguous.

Set field types

For each view Looker has just connected, open the fields panel. Make sure:

  • Any field ending in _pct, pct_, share, or rate is Percent (not Number)
  • Any field ending in _spend, _val, income is Number (currency ZAR)
  • month fields are Date (YYYY-MM)
  • DESTINATION and CATEGORY_TWO are Text and set as dimensions, not metrics

Get this right once per data source — it propagates everywhere.

Set up global filters (do this before Page 1)

Two dropdown controls that apply to every page. This is what makes the dashboard feel like one tool instead of six reports.

Add a Client (DESTINATION) dropdown

Toolbar → Add a control → Drop-down list. Place it top-right of the header band. In the properties panel:

  • Data source: v_dashboard_client_pitch
  • Control field: DESTINATION
  • Label: Client
  • Default selection: Clicks (or any client — you can change at runtime)

Now right-click the control → Make report-level. This is the key step — it makes the filter affect every page, not just this one.

Add a Category (CATEGORY_TWO) dropdown

Same process, field CATEGORY_TWO, label Category. Make it report-level too.

Add a date range control (optional but recommended)

Toolbar → Add a control → Date range control. Default: Last 12 months. Make report-level. Only pages that use v_monthly_trends will respect it — that's fine.

Filter gotcha: Looker Studio auto-applies a control to any chart whose data source contains the field. If a chart uses a view that doesn't have DESTINATION (like v_dashboard_overview), the filter is ignored — that's expected. The Overview page is intentionally FNB-wide.

Overview Page 1 — Executive summary

Scorecard row (5 scorecards across the top)

Data source: v_dashboard_overview. One scorecard per metric:

  • Total transactions → total_transactions
  • Total customers → total_customers
  • Segmented customers → segmented_customers
  • Churn scored → churn_scored
  • Destinations → destinations

Headline bar: selected client snapshot

Below the scorecards, add a single-row table. Data source: v_dashboard_client_pitch. Dimensions: DESTINATION, CATEGORY_TWO. Metrics: customers, total_spend, market_share_pct, avg_share_of_wallet. The report-level filters will narrow this to the selected client × category automatically.

Segment pie (preview)

Add a pie chart. Data source: v_client_segment_mix. Dimension: segment_name. Metric: pct_of_client_customers. Title: "Selected client's customer mix". This is the 10-second hook — already different for every client.

Client Pitch Page 2 — Market position

KPI strip (5 scorecards)

Data source: v_dashboard_client_pitch. Metrics: customers, total_spend, market_share_pct, penetration_pct, avg_share_of_wallet. Report-level filters already scope to client × category.

Competitor bar chart (anonymised)

Add a horizontal bar chart. Data source: v_pitch_external. Dimension: display_name (this is the anonymised label — "Competitor 1", "Market Leader", etc.). Metric: total_spend. Sort descending. Title: "Market position in category".

Use v_pitch_external on any page that will be shown to a client externally. Use v_pitch_internal (real names) on pages that stay inside the team.

Loyalty bands

Add a stacked bar. Data source: v_dashboard_client_pitch. Dimension: DESTINATION. Metrics: band_1_store, band_2_stores, band_3_4_stores, band_5_7_stores, band_8_plus. Shows how concentrated or spread a customer's shops are.

Time-of-day heatmap

Add a pivot table with heatmap. Data source: v_dashboard_client_pitch. Rows: day-of-week (create a calculated dimension CASE peak_day_number WHEN 1 THEN 'Sun' …). Columns: peak_hour. Metric: pct_weekend (or build it from day × hour fields if you want a full heatmap).

Segments Page 3 — Per-client segmentation (the fix)

Read this before you build Page 3. The 5 segments (Champions, Loyal High Value, Steady Mid-Tier, At Risk, Dormant) are defined globally across all FNB customers — a "Champion" means the same thing everywhere. But the mix of segments is different for every client. This page uses v_client_segment_mix so Clicks and PNP show different numbers. Don't connect this page to mart_cluster_output directly — that's the old global table that caused the identical-numbers bug.

Segment distribution (the per-client view)

Add a column chart. Data source: v_client_segment_mix. Dimension: segment_name (sort by segment_order ascending to get Champions first). Metric: pct_of_client_customers. Report-level filter on DESTINATION + CATEGORY_TWO makes this client-specific. Title: "Customer mix for selected client".

Over- / under-index vs FNB

Add a bullet chart or a simple bar. Data source: v_client_segment_mix. Dimension: segment_name. Metric: index_vs_fnb. Reference line at 100. Anything above 100 means this client over-indexes on that segment vs FNB-wide — a pitch hook ("your customers are 1.4× more likely to be Champions than the typical FNB customer").

Segment definitions table

Add a table. Data source: v_dashboard_segments. Dimensions: segment_name, business_description, recommended_action. Metrics: avg_spend, avg_txns, avg_recency_days. This is the reference — what each segment means, identical for every client.

Spend contribution

Add a pie or donut. Data source: v_client_segment_mix. Dimension: segment_name. Metric: pct_of_client_spend. Title: "Where the client's revenue comes from". Often the Champions slice is much bigger here than on the customer-count chart — that's the 80/20 story.

Audiences Page 4 — Audience Marketplace

Scope reminder: The Audience Marketplace is FNB-wide by design. Advertisers buy these pre-packaged audiences (e.g. "High value champions", "Young families 26–40") and activate them via LiveRamp for their own campaigns. The audiences don't change when you change the client filter. What does change is the overlap chart below — "which of these audiences are my customers already in?".

Audience marketplace grid

Add a table. Data source: v_audience_catalog. Dimensions: audience_id, audience_name, audience_type, description. Metrics: audience_size, avg_spend, avg_age, pct_female, avg_income. Add a filter control on audience_type so the user can narrow to Demographic / Lifestyle / Behavioral / Geographic / Seasonal / Cross-category.

Client overlap chart (per-client slice of the marketplace)

Add a horizontal bar. Data source: v_audience_client_overlap. Dimension: audience_name. Metric: pct_of_client. Sort descending. Top N = 10. Title: "Top audiences among {selected client}'s customers". Report-level filter on DESTINATION + CATEGORY_TWO drives this.

This is where the page earns its keep. The grid above says "we have 34 audiences for sale". The overlap chart says "and here's which of them your customers are already in" — that's the activation conversation.

Audience type summary

Add a column chart. Data source: v_audience_catalog. Dimension: audience_type. Metric: SUM(audience_size). Title: "Reach by audience type". Lets a buyer see the scale of each category of audience at a glance.

Churn & CLV Page 5 — Risk & lifetime value

Risk-level scorecards

Data source: v_dashboard_churn. Filter: churn_risk_level IN ('Critical', 'High'). Metrics: COUNT(UNIQUE_ID), SUM(total_spend). Two scorecards side by side.

Segment × risk heatmap

Pivot table. Data source: v_dashboard_churn. Rows: segment_name. Columns: churn_risk_level. Metric: COUNT(UNIQUE_ID). Heatmap style. Quickly shows which segments are bleeding.

CLV tier distribution

Column chart. Data source: v_dashboard_churn. Dimension: clv_tier. Metric: SUM(predicted_clv). Title: "Future revenue by CLV tier".

Top churn reasons

Bar chart. Data source: v_dashboard_churn. Dimension: reason_1. Metric: COUNT(UNIQUE_ID). Top 8. Shows the dominant drivers from the ML model.

Trends & Geo Page 6 — Time & place

Monthly spend trend

Time series. Data source: v_monthly_trends. Dimension: month. Breakdown: DESTINATION. Metric: total_spend. Report-level filters will narrow the DESTINATIONs shown.

Province spend map

Geo chart. Data source: v_geo_insights. Geo dimension: PROVINCE. Metric: total_spend. Zoom: South Africa.

Municipality table

Table. Data source: v_geo_insights. Dimensions: PROVINCE, MUNICIPALITY. Metric: total_spend. Sort descending, top 20.

Anonymisation — internal vs external

Two versions of the same data, one for inside-the-team use and one for showing the client.

ContextUse viewWhat the client sees for competitors
Internal reviewv_pitch_internalReal names ("Dis-Chem", "Pick n Pay")
Client-facingv_pitch_external"Competitor 1", "Market Leader", "Competitor 2"…
Don't mix. Every chart on a page should use the same one. If you put a v_pitch_internal table next to a v_pitch_external chart, you'll leak real names next to anonymised labels and it will be obvious.

Polish checklist

  • Report theme applied (Theme & layout → dark / simple / custom)
  • Page names short and all-caps or title-case (e.g. "CLIENT PITCH", not "client_pitch_page_final_v3")
  • Global filters in the header of every page, visually aligned
  • All currency fields formatted as ZAR with no decimals for values over R1,000
  • Percent fields all formatted as ##.#% with no raw decimals
  • Tables sorted by the most relevant metric, not alphabetically
  • Every page has a one-line description at the top ("Selected client's segment distribution…")
  • No chart titled "Chart 1"
  • File → Make a copy to create a client-specific snapshot when pitching
